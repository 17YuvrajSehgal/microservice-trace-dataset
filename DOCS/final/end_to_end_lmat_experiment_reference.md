# End-to-End LMAT Experiment Reference

## Purpose

This document is the working reference for the full LMAT experiment pipeline used in this repository:

- reproduce the Sock Shop environment
- collect kernel and UST traces under normal and anomaly conditions
- convert raw traces into LMAT training shards
- validate the dataset
- train LSTM and Transformer models
- evaluate OOD anomaly detection performance
- prepare for future online/overhead experiments on a live service

This file is intended to be practical rather than paper-styled. It should be used as the baseline checklist before launching new experiments.

## 1. System Overview

The project studies anomaly detection in Sock Shop traces using LMAT-style sequence models trained on LTTng trace data.

The end-to-end pipeline is:

1. Reproduce and stabilize the Sock Shop microservice environment.
2. Collect raw traces under normal and anomaly scenarios.
3. Export reusable text dumps from raw traces.
4. Preprocess kernel traces into NPZ shards.
5. Validate the generated dataset.
6. Train sequence models.
7. Run OOD evaluation on anomaly splits.
8. Use the trained models for further experiments, including online overhead/latency studies.

## 2. Sock Shop Environment

The traced application is the Sock Shop microservice demo running with Docker Compose plus monitoring components.

Primary environment reference:

- [reproduce_sock_shop_env.md](C:\workplace\adaptive_tracer\DOCS\final\reproduce_sock_shop_env.md)

Important environment assumptions:

- Docker Compose deployment with Sock Shop services and monitoring
- Prometheus available and reachable
- Grafana available for quick VM/service inspection
- LTTng installed and usable for kernel and UST sessions
- `babeltrace` or `babeltrace2` available for reading traces
- `stress-ng` required for CPU, disk, and memory anomaly injection
- `tc` required for network anomaly injection

Practical note:

- `stress-ng` must be installed before running CPU, disk, or memory anomaly scripts.
- Earlier runs collected before confirming `stress-ng` availability should not be trusted as true CPU/disk/memory anomaly data.

### 3.2 What Each Run Produces

Each run produces three categories of outputs:

```text

```

### 3.3 Trace Types

Kernel traces:

- full kernel event capture
- includes syscall entry/exit events
- includes scheduler, timer, RCU, power, and other tracepoints
- includes kernel contexts: `pid`, `tid`, `procname`

UST traces:

- generated by the OTel relay
- contain exported span events from selected services
- useful for cross-layer inspection and documentation
- not used as the primary LMAT training source

### 3.4 Load Generator Settings

Traffic is generated by [load_generator.py](C:\workplace\adaptive_tracer\microservice-lttng-data-collection-scripts\load_generator.py).

Important standardized pacing used across the collection scripts:

- `THINK_MIN=0.1`
- `THINK_MAX=0.3`

Why this matters:

- normal and anomaly runs should use the same pacing defaults
- changing pacing changes workload shape, which can confound anomaly comparisons

### 3.5 Logging and Safety

The collection scripts now write terminal output to `run.log` so experiment behavior can be audited later.

Important safety behavior:

- CPU/disk/memory scripts now fail fast if `stress-ng` is missing
- memory script now fails if `stress-ng` exits immediately
- network script includes bridge detection and cleanup logic

### 3.6 Metrics Collected During Data Collection

`download_metrics.sh` pulls:

VM-level metrics:

- CPU
- memory
- disk
- network receive
- network transmit

Service-level metrics for selected services:

- QPS
- p50 latency
- p95 latency
- errors when available

These metrics are useful for:

- checking whether anomaly injection visibly changed system behavior
- correlating trace behavior with service-level symptoms
- choosing runs worth keeping for later experiments

### 3.7 Important Collection Caveats

These lessons should be carried forward:

- `stress-ng` must be verified before CPU/disk/memory collection.
- Network anomaly injection worked independently because it relies on `tc`, not `stress-ng`.
- The UST relay only captures a subset of services, so UST latency alone should not be treated as the main anomaly signal.
- Raw kernel traces are the authoritative source for LMAT dataset construction.

## 4. Dataset Creation

Primary dataset reference:

### 4.1 Canonical Dataset Used So Far

The main dataset used for the completed experiments is the kernel-centric LMAT dataset with:

- sequence length: `512`
- shard size: `5000`
- paper duration bins: `5`
- effective code-level duration classes: `6`
- event scope: `syscall`

This is the dataset to reference when discussing the completed LSTM and Transformer results up to this point.

### 4.2 Dataset Splits

Default split design:

- `train_id`: `normal/run01`, `normal/run02`, `normal/run03`
- `valid_id`: `normal/run04`
- `test_id`: `normal/run05`
- `valid_ood_cpu`: `anomaly_cpu/run04`
- `test_ood_cpu`: `anomaly_cpu/run05`
- `valid_ood_disk`: `anomaly_disk/run04`
- `test_ood_disk`: `anomaly_disk/run05`
- `valid_ood_mem`: `anomaly_mem/run04`
- `test_ood_mem`: `anomaly_mem/run05`
- `valid_ood_net`: `anomaly_net/run04`
- `test_ood_net`: `anomaly_net/run05`

### 4.3 How the Preprocessor Builds Sequences

The canonical preprocessing behavior is:

- reads raw kernel traces or text dumps
- sorts by timestamp
- uses fixed `100 ms` windows
- groups by thread (`tid`)
- keeps only syscall events in paper-aligned mode
- recovers syscall durations by matching `syscall_entry_*` and `syscall_exit_*`
- learns duration bins from training normal runs only
- writes split-wise NPZ shards

### 4.4 Encoded Feature Channels

Each sequence contains LMAT-style channels:

- `call`
- `entry`
- `duration`
- `proc`
- `pid`
- `tid`
- `ret`
- `lat_cat`

And metadata:

- `seq_len`
- `req_dur_ms`
- `is_anomaly`

### 4.5 Root Dataset Outputs

The preprocessed dataset root contains:

- `vocab.pkl`
- `delay_spans.pkl`
- `dataset_manifest.json`

Each split contains:

- `meta.json`
- `shard_*.npz`

These files are the minimum required artifacts for training and later reproducibility.

### 6.1 Models Trained

Model implementations:

- [LSTM.py](C:\workplace\adaptive_tracer\models\LSTM.py)
- [Transformer.py](C:\workplace\adaptive_tracer\models\Transformer.py)

#### LSTM

The LSTM model:

- embeds the LMAT feature channels
- concatenates them into a single token representation
- uses stacked recurrent layers
- predicts:
  - next syscall token for event modeling
  - duration category for latency modeling

#### Transformer

The Transformer model:

- uses the same LMAT feature embeddings
- applies causal self-attention
- uses autoregressive masking
- predicts the same two heads:
  - event head
  - latency head

### 6.2 Training Modes

All experiments are built from the same shard format and support three regimes:

- event-only
- duration-only
- multi-task

Meaning:

- event-only: train next-syscall prediction only
- duration-only: train latency category prediction only
- multi-task: train both heads jointly

### 6.3 Training Behavior Implemented

Important training features in [train_sockshop.py](C:\workplace\adaptive_tracer\microservice\train_sockshop.py):

- BF16 mixed precision support
- gradient accumulation
- cosine learning-rate schedule with warmup
- validation during training
- best-model checkpointing
- early stopping support
- OOD evaluation after training

Important current behavior:

- multitask loss uses `multitask_lambda`
- early stopping is supported with `--early_stopping_patience`
- after training or early stopping, the script still loads the best checkpoint and runs final OOD evaluation

### 6.4 Training Data Flow

`SockshopNpzDataset` in [NpzDataset.py](C:\workplace\adaptive_tracer\microservice\NpzDataset.py) streams pre-padded mini-batches from NPZ shards and produces:

- model inputs
- shifted targets
- `pad_mask`
- sequence-level metadata

The dataset loader trims each batch to the actual maximum sequence length inside the batch for efficiency.

## 7. Training Outputs

A completed training run typically writes:

- `config.json`
- `metrics.csv`
- `model_best.pt`
- `ood_results.json`
- `train_rank0.log`
- cluster `.out` log

Practical note:

- `train_rank0.log` is the preferred short training log for analysis
- `.out` files can be large and are mainly for full job debugging

### 7.1 Meaning of Each Training Output

`config.json`

- exact training configuration used for the run

`metrics.csv`

- step-wise or validation-wise scalar metrics for later plotting

`model_best.pt`

- checkpoint corresponding to best validation loss

`ood_results.json`

- final OOD evaluation metrics on test splits

`train_rank0.log`

- compact record of model setup, epoch progress, validation events, best checkpoint updates, and final evaluation

## 8. OOD Evaluation Outputs

OOD evaluation is handled in [train_sockshop.py](C:\workplace\adaptive_tracer\microservice\train_sockshop.py) using helpers from [ood_metrics.py](C:\workplace\adaptive_tracer\microservice\ood_metrics.py).

### 8.1 Scores and Metrics

The pipeline computes anomaly scores from:

- event cross-entropy
- latency cross-entropy
- a weighted combination when both heads are used

Reported metrics include:

- AUROC
- AUPR
- F1
- precision
- recall
- accuracy

Threshold selection follows the existing validation-based tuning logic.

### 8.2 Why Validation Splits Matter

Validation ID and validation OOD splits are used to:

- tune OOD thresholds
- determine the best model checkpoint
- support early stopping

This is important for future experiments because threshold tuning should always remain separate from final test reporting.

## 12. Guidance for Upcoming Live LMAT Overhead Experiments

You mentioned the next target is measuring the overhead latency of LMAT on a live service. For that work, the following information should always be captured.

### 12.1 Baseline vs Instrumented Comparisons

For each overhead experiment, compare at least:

- baseline service without LMAT runtime scoring
- service with tracing only
- service with tracing + preprocessing/runtime feature extraction
- service with full LMAT inference enabled

This separation is important because the overhead can come from different layers:

- tracing overhead
- data movement and feature extraction
- model inference overhead
- thresholding and decision logic

### 12.2 Recommended Metrics for Overhead Studies

Record:

- p50 latency
- p95 latency
- p99 latency if possible
- throughput / QPS
- error rate
- CPU usage
- memory usage
- GPU usage if inference is offloaded
- request volume
- sequence extraction rate
- model inference time per request or per sequence

### 12.3 Trace/Inference Metadata to Save

For overhead experiments, also save:

- LMAT model identifier
- checkpoint path used
- duration-bin setting
- sequence length used for inference
- batch size used for inference
- inference device
- whether event, duration, or multitask score is used
- threshold used for anomaly decision

### 12.4 Recommended Practical Checks

Before running live overhead experiments:

1. Confirm the environment is clean and reproducible.
2. Confirm traces and metrics are being written.
3. Confirm `stress-ng` exists if anomaly injection will be reused.
4. Confirm the exact model checkpoint can be loaded.
5. Confirm the scoring path uses the intended head:
   event-only, duration-only, or combined.
6. Confirm logging is lightweight enough not to distort latency.

## 13. Known Lessons and Cautions

- Kernel traces are the primary source for LMAT dataset creation.
- UST traces are useful, but only partial and export-time.
- CPU/disk/memory anomaly scripts depend on `stress-ng`.
- Missing `stress-ng` can silently invalidate anomaly collection if not checked.
- Network anomaly collection is operationally different because it uses `tc`.
- Threshold tuning should always use validation splits, not test splits.
- Completed results should be interpreted relative to the exact dataset version used.

## 14. Minimal End-to-End Checklist

Use this short checklist before any new experiment cycle:

1. Reproduce/reset Sock Shop environment.
2. Confirm Prometheus and tracing dependencies are available.
3. Confirm `stress-ng` and `tc` availability if anomalies will be collected.
4. Run collection and save raw traces plus experiment logs.
5. Export text dumps if preprocessing will be repeated.
6. Build shards with the intended configuration.
7. Validate the dataset.
8. Train models and keep `config.json`, `metrics.csv`, `model_best.pt`, `ood_results.json`, and `train_rank0.log`.
9. Document the result directory and exact script used.
10. For live overhead work, separately capture tracing overhead and model-inference overhead.
